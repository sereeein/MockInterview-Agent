# 面试准备 Agent — 项目立项文档

> 本文档由前期产品策略对话整理而成。目的是把已做出的决策、保留的开放项、被显式排除的方向一次性写清楚，作为后续在 Claude Code 中继续开发的起点。
>
> 建议：放在项目根目录，命名为 `CLAUDE.md` 可作为 Claude Code 的项目记忆自动加载，或保留 `PROJECT.md` 作为人读文档。

---

## 一句话定位

为产品 / 数据 / AI 求职者打造的、基于个人简历反向挖题并能多轮追问的 AI 面试演练 agent；其他岗位求职者也能用（兜底体验）。

---

## 1. 项目背景

### 1.1 开发者画像
- 求职方向：产品（PM / 产品运营 / AI 产品）+ 数据岗（数据科学专业背景）
- 技术起点：调过 OpenAI / Claude API，会基本 prompt 工程；agent 方向首次实践
- 投入预期：1-2 个月，认真做出能用的产品（非周末 demo）
- 简历目的：用作产品 sense + 数据驱动思维的代表性项目

### 1.2 痛点筛选过程
最初候选三个方向，最终筛掉两个：

| 候选方向 | 决定 | 原因 |
|---|---|---|
| 面试准备 | ✓ 选定 | 技术 100% 可控、agent 含量真实、开发者本人就是用户、评估框架清晰 |
| 网申自动填表 | ✗ 砍掉 | 各家系统反爬+DOM 异构，1-2 月单人做不出稳定版；讲故事偏技术，跟产品岗错位 |
| 岗位聚合 | ✗ 砍掉 | 主流招聘平台反爬+法律灰色；本质是 ETL+推荐，不是 agent |

---

## 2. 目标用户与定位

### 2.1 核心人群
产品岗（PM、产品运营、AI 产品）/ 数据岗（数据分析、数据科学、ML 工程）/ AI 产品 / AI 工程岗求职者。

### 2.2 兼容人群
其他岗位求职者：可用，但走通用路径，体验"还行但不惊艳"。

### 2.3 推广渠道
小红书：在「产品经理」「求职」「互联网大厂」「AI 求职」等垂直兴趣池精准切入。

### 2.4 与现有方案的差异化
- 比通用 ChatGPT / Claude 对话强：垂直岗位认知（知道 PM 要考北极星指标、case 题套路；知道数据岗要考 metric 设计、SQL）
- 比通用面试工具（面试鸭、Interview Warmup、Yoodli）强：
  - 基于用户简历**反向挖题**，不是题库匹配
  - **多轮追问深挖**，不是答完就结束
  - 垂直化的评分标准（STAR 完整度判定按岗位特性调整）

---

## 3. 核心差异化策略：「深内核 + 开放表层」

三个候选策略对比后选定：

| 策略 | 描述 | 决定 |
|---|---|---|
| ① 纯垂直 | 只服务 PM/数据/AI，其他岗位拒绝 | ✗ 浪费小红书外溢流量 |
| ② 纯通用 + 岗位路由 | UI 加岗位下拉，后端同一套通用 prompt | ✗ 反馈质量差、小红书无法被识别、讲故事弱 |
| ③ 深内核 + 开放表层 | 三类核心岗位走精细路径，其他岗位走通用兜底 | ✓ 选定 |

### 3.1 选项 ③ 的落地方式
- **前端**：目标岗位选择，"产品 / 数据 / AI"作为主推显著位置 + "其他岗位"作为兜底入口
- **后端两条路径**：
  - 核心三岗位：curated rubrics + 真题种子库 + role-specific drilling patterns
  - 其他岗位：纯 LLM 驱动的通用路径
- **推广 slogan**："专为产品 / 数据 / AI 求职者打造的 AI 面试拷打 agent，其他岗位也能用"

### 3.2 这套策略的简历金句
> "先做 vertical PMF 验证核心循环，再开放表层做 GTM 增长。"

## 4. 四个差异化点的取舍

| 代号 | 名称 | 状态 | 说明 |
|---|---|---|---|
| A | 垂直定位 | v1 必做 | 几乎不抢工时，是 prompt 规约 + 种子库 curate，不是独立功能模块 |
| C | 简历反向挖题 | v1 核心 | 用户输入简历，agent 反推面试官最可能挖的角度并 STAR 化预答 |
| D | 多轮追问 | v1 核心 | 顺着用户回答深挖 3 层，是 agent 循环价值的最佳展示 |
| B-reframed | 候选人情报 | v1.5 增强 | 聚合 Glassdoor、看准网、一亩三分地、Reddit、知乎等"前人面经" |

### 4.1 关键耦合
**C + D 是耦合的**：C 负责出题、D 用 C 的题为基础往深挖。这两件事必须一起做，否则核心循环就没有灵魂。

### 4.2 为什么 B-reframed 留到 v1.5
- 候选人情报本身是一个独立的工程坑（反爬 + 数据清洗 + 时效排序 + 可信度排序）
- 如果 v1 就背上它，4 周后可能还卡在数据层，核心引擎一行没写
- 引擎稳定后，B 只是给出题阶段挂一个新的数据源，下游逻辑不用改

### 4.3 关于 B-reframed 的明确定义
区分两种 B：
- ❌ **原版 B**（已弃）：抓公司新闻 / 财报 / 高管访谈，让 agent 看新闻
- ✅ **B-reframed**（v1.5 采用）：抓求职者 / 在职员工 / 离职员工的评价、面试经验、对公司偏好特质的反馈，让 agent 做信息考古

---

## 5. v1 / v1.5 范围拆分

### 5.1 v1（前 4 周）核心引擎
- 实现：A + C + D
- 输入：简历 / JD / 公司名 / 目标岗位
- 输出：个性化题库 + 演练记录 + 改进点
- 目标：跑通核心循环，开发者本人每次面试都拿来用

### 5.2 v1.5（后 4 周）增强层
- 接入 B-reframed 候选人情报
- 在出题阶段加一个数据源检索步骤
- 风险低：引擎已稳定，B 只是更好的输入

### 5.3 拆法的关键好处
1. v1 失败仍有可交付项目（核心引擎独立成立）
2. v1 成功有清晰升级路径，简历叙事节奏好

## 6. Agent 工作流架构

### 6.1 数据流图

```
   ┌──────┐  ┌─────┐  ┌──────────────┐
   │ 简历 │  │ JD  │  │ 公司 + 岗位  │
   └──┬───┘  └──┬──┘  │ A: PM·数据·AI│
      │         │     └──────┬───────┘
      └─────────┼────────────┘
                ▼
      ┌──────────────────┐       ┌────────────────┐
      │   出题引擎       │ ◀╌╌╌╌│ 候选人情报      │
      │ C: 简历反向挖题  │  v1.5 │ B (v1.5 才接)   │
      └────────┬─────────┘       └────────────────┘
               │
               ▼
      ╭───────────────────────────╮
      │   Agent 提问 / 追问        │ ◀──┐
      │           ↓                │     │
      │      用户回答              │     │ D 多轮追问
      │           ↓                │     │ (不完整时回流)
      │      STAR 完整?            │     │
      │      ├─不完整 ─────────────┘
      │      └─完整 ─→ 下一题/输出
      ╰───────────────────────────╯
               │
               ▼
   ┌──────────────────────────┐
   │ 题库 + 演练记录 + 改进点 │
   └──────────────────────────┘
```

### 6.2 两个核心认知
1. **右侧的 U 型循环**（提问→回答→STAR 检查→不完整就追问）是 D 的灵魂，也是这个 agent 区别于普通对话工具的核心。
2. B 在 v1.5 接入时只挂在出题引擎旁，不影响下游逻辑——这是良好架构带来的低成本迭代能力。

## 7. 评估指标（待阶段 2 落地）

数据岗 + 产品岗的简历叙事都需要"我怎么证明它好用"。初步思路：

- **题目相关性**：LLM-as-judge 评估生成题目和 JD/简历的契合度 + 人工抽检
- **追问深度**：平均能追到第几层、有没有命中真实面试官的高频追问点
- **用户自评**：感觉像不像真实面试（NPS 量表）
- **种子用户 metric**：开发者本人参加 N 次真实面试后，统计预演命中率（被问到的题中 agent 提前练过的占比）——这是简历上最有说服力的数据
- **对比基线**：同样输入下与裸 ChatGPT / Claude 对话的体验对比

---

## 8. 明确的非目标（不做什么）

- ✗ 不做网申自动填表
- ✗ 不做岗位聚合 / 招聘信息爬虫
- ✗ v1 不接入候选人情报数据层（留 v1.5）
- ✗ 不为非核心岗位（PM/数据/AI 之外）做精细化体验，只走通用兜底
- ✗ 不做语音 / 视频面试模拟（v1.x 都不做，未来可考虑）
- ✗ 不做账号体系（v1 单机即可，无需登录）

---

## 9. 已知前提与约束

- 开发者：1 人、1-2 个月、agent 经验为零、调过 LLM API
- 数据来源：用户输入（简历、JD）+ 公开网页（公司信息 / 面经，v1.5 才接）
- v1 不引入需要长期维护的外部系统
- 主开发环境：Claude Code

## 10. 阶段 2 开放问题（在 Claude Code 中继续设计）

阶段 2 目标：把 v1 从概念落到可执行。需要回答：

1. **用户故事 & 核心使用流程**：1 页纸的"我是谁、什么场景下、怎么用、得到什么"
2. **每个 agent 步骤的 I/O 与 prompt**
   - 出题引擎：怎么从简历项目反推问题、覆盖 JD 关键能力
   - STAR 检查：怎么判定回答是否完整
   - 追问生成：怎么挑回答里最弱的一环深挖
3. **技术选型**
   - agent 实现：原生 function calling / LangChain / LangGraph / 手写循环？
   - LLM：Claude / GPT / 双模型分工？
   - 前端：CLI / Streamlit / Next.js？影响小红书演示呈现
   - 持久化：SQLite / JSON / 不持久化？
4. **评估方法落地**：搭一个最小评估集（10-20 道真题 + 真简历）做回归测试
5. **4 周开发日历**：每周做什么、周末有什么可演示成果
